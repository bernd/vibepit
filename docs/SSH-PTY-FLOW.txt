  Two things worth calling out:                                                                                                                                                           
                                                                                                 
  1. The VTE is on a side branch of the output pump, not in series. Live clients get the raw PTY bytes unmodified (session.go:304). The VTE is only read on attach to synthesize a replay 
  (session.go:138-184). So if nobody ever re-attaches, the VTE is basically just absorbing the output stream for nothing — it's the re-attach path that justifies it.
  2. Terminal queries get answered by the user's real terminal, not the VTE. When an app writes ESC[6n, those bytes flow to both the VTE and to the attached client's SSH channel. The    
  user's real terminal emulator sees the query in its output stream, responds on its input stream, and the response goes back through SSH → WriteInput → PTY → app stdin. The VTE's own   
  response goes into drainVTE and is thrown away (session.go:272). That means with no client attached, TUI queries go unanswered — the drain only exists to keep vte.Write from blocking
  on a full response pipe.
